## 1. Why this milestone exists

User ran a council debate session in project `D:\sources\eBerth` with topic:

> "hiện tại tôi đang run ứng dụng ở http://localhost:3010/planning bạn reading các file trong folder docs/* để hiểu nghiệp vụ sau đó đi vào url http://localhost:3010/planning để research bàn luận đánh giá hướng phát triển tiếp theo (full focus ui only và full focus workspace planning) nhớ research đối chiếu thêm ở internet nữa nhé"

User expected the council to:
1. Read `docs/*` files for business context
2. Navigate to `http://localhost:3010/planning` to inspect the running UI
3. Cross-check with internet research
4. Have agents really debate (not 2 monologues)
5. Produce evidence-grounded recommendation

**What actually happened (per `[Council Memory]` record in DB):**
- Debate-planner LLM call FAILED → fallback to generic stances `Primary Analyst` / `Critical Reviewer` (instead of domain-specific roles like `UI Architect` / `Berth Ops Expert`)
- `stats.calls = 0` in saved record (accounting bug — see §3)
- `finalPositions = ["", ""]` for both roles (mutation bug — see §3)
- Synthesis was generated but `parseOutcome` failed → no `[Council Decision]` / `[Council Outcome]` records → no action plan
- Synthesis content was structurally correct (Agreed / Tradeoffs / Recommendation) but **had zero evidence citations** — no file paths, no line numbers, no URL screenshots, no network responses; model itself flagged "Drag-and-drop gain of 40% is based on external studies and unverified"
- Debate phase ran 502s but research role couldn't actually do what user asked

**Quality scoring of synthesis:**

| Dimension | Score | Reason |
|---|---|---|
| Structure | 8/10 | Correct output shape |
| Topic relevance | 7/10 | Used right vocabulary (berth, tide, dnd-kit) |
| Evidence-grounded | **2/10** | No citations, model self-admitted speculation |
| Met user request | **3/10** | Didn't read docs/, didn't visit URL, didn't research internet |
| Actionable | 5/10 | Had recommendation but `outcome=null` blocked execution |

**Root cause:** the council architecture as of 2026-05-08 has 5 structural problems that make evidence-grounded multi-agent debate impossible. v1.6 fixes these in 4 sequential phases.

## 2. Architecture as of audit

### 2.1 Council flow (current)

```
runCouncil (council/index.ts)
  ├─ resolveLeader / resolveParticipants (council/leader.ts)
  ├─ buildCouncilContext (council/context.ts)
  ├─ Phase A — runClarification (council/clarifier.ts)            ← LLM call
  ├─ Phase A — runPreflight (council/preflight.ts)                 ← user confirm
  ├─ Phase B — planDebate (council/debate-planner.ts)              ← LLM call (FAILED in audit)
  ├─ Phase C — runDebate (council/debate.ts)
  │     ├─ evaluateResearchNeed (LLM)
  │     ├─ llm.research()  ← codebase tools only (bash/grep/read_file)
  │     ├─ Phase 1: parallel openings (text-only llm.generate)
  │     ├─ Phase 2: discussion rounds (text-only llm.generate, max 8)
  │     │   ├─ pair exchanges
  │     │   ├─ leader evaluateDebate (LLM)
  │     │   ├─ optional mid-research
  │     │   └─ inter-round summary (LLM)
  │     └─ returns DebateState { exchangeLogs, runningSummary, roundCount }
  ├─ Phase D — runPlanning (council/planner.ts)
  │     ├─ buildSynthesisPrompt → tracedGenerate (LLM)
  │     ├─ parseOutcome (regex/JSON)                                ← FAILED in audit
  │     └─ optional preflight-confirmed action plan
  ├─ Persist [Council Decision] / [Council Outcome] / [Council Memory] (storage)
  └─ Phase E — runExecution (council/executor.ts) if plan approved
```

### 2.2 PIL (Prompt Intelligence Layer) — NOT integrated with council

`src/pil/pipeline.ts:runPipeline` is invoked from `processMessage` (orchestrator) for the MAIN agent's system prompt construction. **Council bypasses PIL entirely.**

6 layers:
1. `layer1-intent.ts` — taskType classification (regex → tree-sitter → EE brain fallback)
2. `layer2-personality.ts` — personality directives based on developer profile
3. `layer3-ee-injection.ts` — inject relevant EE experiences as warnings
4. `layer4-gsd.ts` — inject GSD workflow context (resume digest, gray areas)
5. `layer5-context.ts` — file/symbol context enrichment
6. `layer6-output.ts` — task-type-specific output style suffix

PIL output: enriched `PipelineContext` with `taskType`, `complexityTier`, `grayAreas`, `domain`, `outputStyle`. Council currently uses ONLY `grayAreas` (via `getPilLastResult` in `council/index.ts:84`) and only for clarifier seeding — drops everything else.

### 2.3 EE (Experience Engine) — NOT integrated with council

`src/ee/*` — communicates with remote brain. **Mode in v1.6 scope: thin-client only.** EE runs on VPS at `experience.muonroi.com` (`~/.experience/config.json` `version: "thin-client"`, auth via `serverAuthToken` / `serverReadAuthToken`). Fat-mode (`~/.experience/experience-core.js`) is OUT OF SCOPE — that file is absent on the dev box and bringing it up is a separate project.

Current usage:
- `ee/bridge.ts:classifyViaBrain` — used by PIL Layer 1, Layer 6
- `ee/intercept.ts:checkBeforeTool` — used by PreToolUse hooks for main agent's Edit/Write/Bash
- `ee/judge.ts:judgeOutcome` — exists but only invoked by phase-tracker, not council
- `ee/mistake-detector.ts` — pattern matching on tool outputs in main flow
- `ee/phase-outcome.ts:recordPhaseOutcome` — records to brain after GSD phase completes

Council does NOT call ANY EE function. It does not query past warnings, does not check tools against experience, does not record outcomes for learning.

### 2.4 MCP infrastructure — exists but not wired to council research

`src/mcp/runtime.ts:buildMcpToolSet` builds a `ToolSet` from configured MCP servers. Catalog (`src/mcp/catalog.ts`) includes:
- `playwright` — browser automation
- `chrome-devtools` — browser inspection
- `tavily` — internet search (with TAVILY_API_KEY hydration from keychain)
- `context7` — library docs
- `filesystem` — file ops

`council/llm.ts:research()` constructs `researchTools = createBuiltinTools(bash, mode)` — **only built-in tools (bash/read_file/grep), no MCP**. So even when user explicitly asks for internet research or URL navigation, the research role technically cannot do it.

## 3. Bugs & gaps identified

### 3.1 Bug — `stats.calls` always 0 in council memory (P0)

**File:** `src/council/index.ts:43`
```ts
const stats: CouncilStats = { calls: 0, startMs: Date.now(), phases: [] };
```

**File:** `src/orchestrator/orchestrator.ts:2049`
```ts
const councilStats = { calls: 0, startMs: Date.now(), phases: [] };
const llm = createCouncilLLM(this.bash, this.mode, this.session?.id, councilStats);
```

`llm.generate` (`src/council/llm.ts:29`) increments `councilStats.calls` (the orchestrator's object). But the persisted record at `src/council/index.ts:227` reads `stats.calls` (council/index.ts's local object that nothing increments). **Result:** every council memory record has `calls: 0` — impossible to audit API usage.

### 3.2 Bug — `finalPositions` always empty (P0)

**File:** `src/council/index.ts:73`
```ts
const active: CouncilParticipant[] = participants.map((p) => ({ ...p, position: "" }));
```

**File:** `src/council/debate.ts:114, 209-210`
- `runDebate` receives `participants` (from index.ts's `active`), creates its OWN local `active` array (line 28), pushes openings into it (line 114), and mutates `position` on local items (lines 209-210)
- Local mutations DO NOT propagate back to `index.ts`'s `active`

**File:** `src/council/index.ts:225`
```ts
finalPositions: active.map((a) => ({ role: a.role, position: a.position.slice(0, 1000) })),
```

Reads `index.ts`'s `active` — positions still `""` from line 73.

### 3.3 Bug — `parseOutcome` fragile (P1)

**File:** `src/council/planner.ts:64`
```ts
outcome = parseOutcome(synthesisText, debatePlan);
```

Implementation uses regex/JSON extraction. When LLM returns text that doesn't exactly match expected shape, returns `null` — and there's no logging of the raw text, no fallback parser using `debatePlan.outputShape.sections` (which IS available and SHOULD be the source of truth for parsing).

In audit session: synthesis was generated correctly with `---READABLE---` separator, but `[Council Decision]` / `[Council Outcome]` records are missing → `outcome === null` → action plan never built.

### 3.4 Gap — research role lacks MCP tools (P0)

**File:** `src/council/llm.ts:39`
```ts
const researchTools = createTools(bash, mode);  // builtin only
```

User asked for internet research + URL navigation; research role cannot do either. `buildMcpToolSet` exists but isn't called here.

### 3.5 Gap — discussion rounds are text-only (P1)

**File:** `src/council/debate.ts:90, 173, 183, 194, 204`

All opening/response/followup calls go through `llm.generate(modelId, system, prompt)` which does NOT pass tools. Agents cannot verify each other's claims by grep'ing code, fetching URLs, or checking files. Result: 2 LLMs trade prose; nobody refutes with evidence.

### 3.6 Gap — debate-planner falls back too eagerly (P2)

**File:** `src/council/debate-planner.ts:43-46`

Single try with regex JSON extraction. If LLM returns markdown-wrapped JSON or extra prose, `match[0]` may be malformed → falls back to generic stances. Should either retry with "your previous output was invalid, here's the schema" or use AI SDK's structured output mode.

### 3.7 Gap — PIL & EE not integrated with council (P1)

See §2.2 and §2.3. Council does not benefit from intent classification, complexity tier, EE experience warnings, EE judge scoring, or phase-outcome learning.

## 4. Phase breakdown for v1.6

### Phase 14 — Council Accounting & Research MCP Wiring (P0, ~1 week)

**Goal:** Make council outputs auditable AND make research role actually capable of internet/URL/code research.

**Success criteria:**
1. Council memory records show non-zero `stats.calls` matching actual LLM call count
2. `finalPositions` in council memory reflect each agent's actual end-of-debate position
3. When `researchTools` resolves, MCP tools (tavily, playwright, chrome-devtools, filesystem) are merged with builtin tools
4. When topic contains an `https?://` URL, research role MUST call a Playwright/Chrome-DevTools tool at least once
5. Research output template enforces 3 sections: `## Source Code Findings` / `## Internet Findings` / `## Frontend Findings (live)` with citations
6. Acceptance: rerun audit topic → council memory has non-empty positions AND research findings cite ≥1 file path AND ≥1 URL AND ≥1 browser snapshot

**Tasks:** 1.1–1.6 from plan draft. ~6 tasks (3 S, 3 M).

**Depends on:** none.

### Phase 15 — Tool-grounded Debate Rounds (P1, ~1 week)

**Goal:** Agents actually debate by verifying each other's claims with tools, not just trade prose.

**Success criteria:**
1. Opening / response / followup calls support `tools` parameter (MCP + builtin merged)
2. Stance prompts mandate tool verification when opponent makes verifiable claims; refutations tagged `[REFUTED via <tool>:<evidence>]`
3. Leader `evaluateDebate` adds `evidenceDensity` (citations / claims) and `disagreementResolved` metrics
4. Each round persisted to DB as `[Council Round N]` system message with citations
5. Debate-planner uses structured output (JSON schema) instead of regex parsing
6. Acceptance: contentious topic ("Redis vs Postgres LISTEN/NOTIFY") logs ≥1 `[REFUTED]` instance or explicit concession

**Tasks:** 2.1–2.6 from plan draft. ~6 tasks (1 L, 4 M, 1 S).

**Depends on:** Phase 14 (research wiring is reused for round tools).

### Phase 16 — PIL + EE Integration into Council (P1, ~1 week)

**Goal:** Council benefits from intent classification, complexity tiers, past experiences, and feeds learning back to brain.

**Success criteria:**
1. `runCouncil` invokes `runPipeline(topic)` early; propagates `taskType`/`complexityTier`/`domain` to debate-planner
2. New `ee/council-bridge.ts:queryExperience(topic, domain)` returns relevant EE warnings
3. Debate-planner injects experience snippets into stance generation prompt
4. When EE returns ≥1 high-confidence warning, an "Experience Auditor" stance is auto-added
5. Tools used in rounds wrapped with `wrapToolWithEeCheck` → PreToolUse warnings stream into debate output
6. After synthesis, `ee/judge.ts:judgeOutcome` scores confidence ∈ [0,1]; if < 0.5 → leader forces another round or `[NEEDS HUMAN REVIEW]` flag
7. `ee/phase-outcome.ts:recordCouncilOutcome` pushes synthesis + verdict to brain
8. Layer 6 output style (`ctx.outputStyle`) respected in synthesis (concise/balanced/detailed)
9. Feature flag `council.experienceMode = off|advisory|enforcing` for safe rollout
10. Acceptance: topic with known brain warning produces "Experience Auditor" stance + warnings cited in rounds; offline mode (EE down) does not crash

**Tasks:** 3.1–3.9 from plan draft. ~9 tasks.

**Open questions for plan-phase to resolve:**
- EE thin-mode latency budget on critical path of debate (target ≤500ms cumulative)
- `ee/judge.ts` schema fit — may need new dimension for council outcomes
- Default value for `council.experienceMode` flag (recommend `advisory` initially)

**Depends on:** Phase 15 (round tool tracing is the integration point for `wrapToolWithEeCheck`).

### Phase 17 — Council Robustness & Observability (P2, ~1 week)

**Goal:** Self-auditable council, user can inspect any past session, doctor catches missing config.

**Success criteria:**
1. `parseOutcome` logs raw synthesis on failure + tries shape-based fallback parser using `debatePlan.outputShape.sections`
2. Debate-plan JSON validation with 1-shot retry before falling back to generic
3. New `/council inspect <session-id>` slash command renders council memory with citations, per-agent tool calls, evidence density
4. Every tool call in research/rounds persisted as `[Council Tool Trace]` system message
5. `muonroi doctor` warns when MCP tavily/playwright not enabled but user has run debates with URL/research-keyword topics ≥3 times
6. README + new `docs/Council.md` document the integrated flow (PIL → EE warnings → planner → debate with tools → EE judge → synthesis)
7. E2E test: rerun audit topic; assert evidence from docs/* + Tavily + Playwright snapshot of localhost

**Tasks:** 4.1–4.7 from plan draft. ~7 tasks.

**Depends on:** Phase 16.

## 5. Risks & mitigations

| Risk | Impact | Mitigation |
|---|---|---|
| MCP servers slow → research timeout | Phase 14/15 | Per-tool timeout (10s default) + parallel-safe gather; report partial findings |
| EE thin-mode adds latency | Phase 16 | `council.experienceMode=advisory` default; bypass on `=off` |
| Structured output not supported by all providers | Phase 15 | Detect provider capability; fall back to regex with retry |
| Persisted `[Council Tool Trace]` blows up DB size | Phase 17 | Truncate tool args/results to 2KB each; rotate after N entries per session |
| User runs council in repo without git/files | Phase 14 | Research falls back gracefully if codebase tools unavailable; reports gap |

## 6.5 EE state verification (2026-05-08, post-milestone-creation)

After milestone v1.6 was created, a follow-up audit confirmed EE state on the dev box. Key findings drove the addition of CQ-16a..d to Phase 16:

- ✓ **EE thin-client healthy** — `~/.experience/config.json` `version: "thin-client"`; `serverBaseUrl=https://experience.muonroi.com`; `curl -H "Authorization: Bearer <readToken>" /health` → 200 with `qdrant=ok, fileStore=ok, embed=unknown, uptime≈47944s`
- ✓ **Intercept loop alive** — 562 `ee_intercept` events today (last 08:38), cumulative 2208 `ee_judge`. Auth, circuit breaker, and offline queue all wired and functioning.
- ✗ **Layer 3 injection 0% success rate** — `interaction_logs WHERE event_type='ee_injection'` last 30 days: 27 `no_match` + 24 `error` (older sessions when VPS was unreachable) + 4 `filtered_noise` (below score floor 0.55) + **0 `injected`**. Today: 6 no_match + 4 filtered_noise, no `error` → server now reachable but brain has no relevant points for user prompts. Score floor `0.55` filters loosely-related material; brain bootstrap is needed (more `experience extract` / `experience evolve` cycles).
- ✗ **UI render sink unwired** — `src/ee/render.ts` defaults `_sink = console.warn`; `setRenderSink` is never called outside tests (`grep src/ui src/index.ts`). In TUI raw mode, `console.warn` writes to stderr which Ink suppresses. PreToolUse warnings fire but the user never sees them.
- ✓ **Layer 3 IS read by model when injection succeeds** — confirmed by trace: `layer3-ee-injection.ts:148` appends `[experience: ...]\n- <text> [id:xxxx]` to `ctx.enriched`, then `orchestrator.ts:2946` reads `pilCtx.enriched` as the user message sent to the model. Once data + render are unblocked, the integration just works.

**Implication for Phase 16:**
- `runCouncil → queryExperience` integration logic is necessary but not sufficient — even after wiring, the user will see nothing if `setRenderSink` stays unwired (CQ-16a) and won't know which experience was applied (CQ-16b).
- Doctor checks for thin-client health (CQ-16c) and brain emptiness (CQ-16d) make these problems discoverable instead of silent.
- **Fat-mode is OUT OF SCOPE for v1.6.** All Phase 16 design assumes thin-client only. The thin-client circuit breaker + offline queue handle VPS unreachability.
