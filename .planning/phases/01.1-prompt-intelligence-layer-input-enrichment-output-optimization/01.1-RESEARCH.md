## User Constraints (from CONTEXT.md)

### Locked Decisions

- **D-01:** PIL intercepts in orchestrator.ts immediately after `UserPromptSubmit` hook fires (line 1993) and before `userMessage` is pushed to `this.messages` (line 1996). The enriched string replaces the raw `userMessage` in the `ModelMessage`.
- **D-02:** Module lives at `src/pil/` — short, consistent with `src/ee/`, `src/router/`, `src/gsd/`.
- **D-03:** Pipeline is a sequential chain of async functions in `src/pil/pipeline.ts`. Each layer receives a `PipelineContext` object and returns it mutated (or unchanged on error).
- **D-04:** Fail-open on two levels: (1) each layer is wrapped in `try/catch` and returns the context unchanged on any error; (2) the entire pipeline runs inside `Promise.race([pipeline(ctx), timeout(200)])` — if 200ms is exceeded the original raw prompt proceeds with no enrichment.
- **D-05:** Reuse `src/router/classifier/` (already built for in-process hot-path classification at <1ms). Extend with PIL-specific task type mapping rather than building a new classifier.
- **D-06:** Six task types: `refactor | debug | plan | analyze | documentation | generate`. `null` = conversational/unclear → Layer 6 passthrough, no output optimization.
- **D-07:** Layer 1 latency budget: 50ms. On breach, `taskType = null` (fail-open).
- **D-08:** Typed context object: `PipelineContext { raw: string; enriched: string; taskType: TaskType | null; domain: string | null; layers: LayerResult[]; }`. `LayerResult = { name: string; applied: boolean; delta: string | null; }` for `/optimize` output.
- **D-09:** Layers 2 (Personality), 3 (EE Injection), 4 (GSD Structuring), 5 (Context Enrichment) ship as pass-through stubs in this phase. Each stub logs `{ name, applied: false }` in `layers[]`.
- **D-10:** Enforcement via system prompt suffix appended to the existing `buildSystemPrompt()` output in orchestrator.ts. Suffix is task-type conditioned: coding tasks get structured output rules. Conversational turns (taskType = null) get no suffix.
- **D-11:** Output optimization applies when `taskType !== null` only. Passthrough for pure conversational turns.
- **D-12:** Token measurement: add `pil_active: boolean` and `enrichment_delta: number` fields to the `UsageEvent` emitted via `recordUsageEvent()`. Delta = estimated output token savings. Visible in `/cost`.
- **D-13:** Safety guardrails in this phase: simple forbidden-patterns list injected into the suffix — prevents "I cannot", "As an AI", "I apologize", looping apologies. Complex adversarial safety deferred to Phase 4 EE.
- **D-14:** Lives at `src/ui/slash/optimize.ts`, registered via `registerSlash("optimize", handler)` — same pattern as `cost.ts`, `route.ts`.
- **D-15:** `SlashContext.lastPrompt?: string` is populated with the enriched prompt after each PIL pass. `/optimize` reads this field.
- **D-16:** No-arg form (`/optimize`) prints the last turn's layer-by-layer summary table: layer name, applied (yes/no), delta snippet. Arg form (`/optimize "custom prompt"`) runs the PIL pipeline on the given string and prints the same table.
- **D-17:** Output format: plain-text table (terminal-compatible, headless-safe). No ANSI color required.

## Summary

Phase 01.1 builds a thin in-process pipeline (`src/pil/`) that runs between the `UserPromptSubmit` hook and the `messages.push()` call in `orchestrator.ts`. The entire pipeline fits within the existing architecture: Layer 1 reuses the already-built `classify()` function from `src/router/classifier/`, Layer 6 appends a conditional string suffix to the existing `buildSystemPrompt()` return value, and the `/optimize` slash command follows the exact same self-registration pattern as `/cost` and `/route`.

The most complex integration concern is the `SlashContext.lastPrompt` write path — the orchestrator currently does not hold a reference to `SlashContext`. This needs a store-level solution (write to a module-level store, then slash context reads it) rather than direct parameter passing. The DB schema migration for `usage_events` is straightforward (migration v3 adding two nullable INTEGER columns with DEFAULT 0).

**Primary recommendation:** Implement in three waves — (1) `src/pil/` module with types + pipeline + L1 + L6 + all stubs; (2) orchestrator integration (intercept + suffix injection + SlashContext write); (3) `/optimize` slash command + DB migration + `/cost` display update.

### Pattern 1: PipelineContext Type Definition

**What:** Typed context object threaded through all six layers.
**When to use:** All layer functions receive and return this.

```typescript
// src/pil/types.ts
export type TaskType = 'refactor' | 'debug' | 'plan' | 'analyze' | 'documentation' | 'generate';

export interface LayerResult {
  name: string;
  applied: boolean;
  delta: string | null;
}

export interface PipelineContext {
  raw: string;
  enriched: string;
  taskType: TaskType | null;
  domain: string | null;
  layers: LayerResult[];
}
```

### Pattern 2: Layer Function Contract

**What:** Each layer is `async (ctx: PipelineContext) => PipelineContext` wrapped in try/catch.
**When to use:** All six layers follow this contract.

```typescript
// Fail-open layer wrapper pattern (from EE PostToolUse fire-and-forget precedent)
export async function someLayer(ctx: PipelineContext): Promise<PipelineContext> {
  try {
    // ... transformation
    return { ...ctx, layers: [...ctx.layers, { name: 'layer-name', applied: true, delta: '...' }] };
  } catch {
    return { ...ctx, layers: [...ctx.layers, { name: 'layer-name', applied: false, delta: null }] };
  }
}
```

### Pattern 3: Sequential Pipeline with Timeout

**What:** `Promise.race` wrapping the full pipeline chain with a 200ms timeout.
**When to use:** `runPipeline()` in `src/pil/pipeline.ts`.

```typescript
// src/pil/pipeline.ts
import { layer1Intent } from './layer1-intent.js';
import { layer2Stub } from './layer2-stub.js';
// ... other layers

async function runLayers(ctx: PipelineContext): Promise<PipelineContext> {
  ctx = await layer1Intent(ctx);
  ctx = await layer2Stub(ctx);
  ctx = await layer3Stub(ctx);
  ctx = await layer4Stub(ctx);
  ctx = await layer5Stub(ctx);
  ctx = await layer6Output(ctx);
  return ctx;
}

export async function runPipeline(raw: string): Promise<PipelineContext> {
  const ctx: PipelineContext = { raw, enriched: raw, taskType: null, domain: null, layers: [] };
  try {
    return await Promise.race([
      runLayers(ctx),
      new Promise<PipelineContext>((resolve) =>
        setTimeout(() => resolve(ctx), 200)  // timeout returns untouched ctx
      ),
    ]);
  } catch {
    return ctx; // total fail-open: return unmodified context
  }
}
```

### Pattern 6: Layer 6 — System Prompt Suffix

**What:** Task-type-conditioned suffix appended to `buildSystemPrompt()` return value.
**When to use:** `layer6-output.ts` + new `applyPilSuffix()` function called in orchestrator.

```typescript
// src/pil/layer6-output.ts
const OUTPUT_SUFFIX_CODING = `

OUTPUT RULES (strict):
- Be extremely concise. No filler, no preamble, no meta-commentary.
- Use bullet points and code blocks. Prefer tool calls over prose.
- Never say "I cannot", "As an AI", "I apologize", or start with "Certainly".
- If a code change is needed, output only the diff or the changed function.
- Prefer structured output (Tool Calling) over free-form text for all coding tasks.`;

export function applyPilSuffix(systemPrompt: string, ctx: PipelineContext): string {
  if (ctx.taskType === null) return systemPrompt;
  return systemPrompt + OUTPUT_SUFFIX_CODING;
}
```

### Pattern 8: PIL Last-Result Store (for /optimize)

**What:** Module-level store that holds the `PipelineContext` of the last turn.
**When to use:** Orchestrator writes after each pipeline run; `/optimize` slash command reads.


### Pattern 9: /optimize Slash Command

**What:** Reads `getPilLastResult()` or runs pipeline on arg string; prints table.
**When to use:** `src/ui/slash/optimize.ts`.

```typescript
// src/ui/slash/optimize.ts
import { registerSlash } from './registry.js';
import type { SlashHandler } from './registry.js';
import { runPipeline } from '../../pil/index.js';
import { getPilLastResult } from '../../pil/store.js';

export const handleOptimizeSlash: SlashHandler = async (args, _ctx) => {
  const ctx = args.length > 0
    ? await runPipeline(args.join(' '))
    : getPilLastResult();
  if (!ctx) return '/optimize: no prompt processed yet — run a prompt first.';

  const header = `Enriched prompt: ${ctx.enriched}\n\nLayer breakdown:`;
  const table = ctx.layers.map(l =>
    `  ${l.name.padEnd(28)} applied=${l.applied ? 'yes' : 'no '}  delta=${l.delta ?? '(none)'}`
  ).join('\n');
  return `${header}\n${table}`;
};

registerSlash('optimize', handleOptimizeSlash);
```

### Pattern 10: DB Migration v3 for PIL Fields

**What:** Add `pil_active INTEGER NOT NULL DEFAULT 0` and `enrichment_delta INTEGER NOT NULL DEFAULT 0` to `usage_events` via `ALTER TABLE` (SQLite supports adding columns).
**When to use:** `src/storage/migrations.ts` — bump `LATEST_DB_VERSION` to 3, add migration branch.

```typescript
// In applyMigrations():
if (version < 3) {
  db.exec(`
    ALTER TABLE usage_events ADD COLUMN pil_active INTEGER NOT NULL DEFAULT 0;
    ALTER TABLE usage_events ADD COLUMN enrichment_delta INTEGER NOT NULL DEFAULT 0;
  `);
  db.pragma('user_version = 3');
}
```

**IMPORTANT:** SQLite's `ALTER TABLE ... ADD COLUMN` does not support STRICT mode restrictions on the new column when the table was created with STRICT. The column must have a DEFAULT. Use `INTEGER NOT NULL DEFAULT 0` — this is valid even with STRICT tables.

### Anti-Patterns to Avoid

- **Calling classify() from inside Layer 1 with async/await and no timeout guard:** The classifier is synchronous (<1ms) but the 50ms L1 budget must still be enforced at the layer wrapper level to match the D-07 contract.
- **Importing EE client or any network module from `src/pil/`:** The existing arch test `tests/arch/no-network-in-classifier.test.ts` checks `src/router/classifier/` only. However, to keep PIL hot-path clean, avoid importing `src/ee/` in Layer 1. EE imports belong in Layer 3 (stub for now).
- **Mutating `ctx.layers` array directly:** Always spread `[...ctx.layers, newResult]` to keep immutable snapshot per layer — needed for `/optimize` debug reproducibility.
- **Calling `buildSystemPrompt()` twice:** Call it once, then pass the result to `applyPilSuffix()`. Do NOT wrap `buildSystemPrompt()` internally.
- **Using `setTimeout` returning a resolved `PipelineContext` for the timeout branch:** The timeout must resolve with the *original* `ctx` captured at pipeline start (before any layers ran), not the in-flight mutated copy. Capture `ctx` before `runLayers()`.
- **Adding `pil_active` / `enrichment_delta` to the `recordUsageEvent()` function signature without adding defaults:** Existing callers pass positional args — either add them as optional trailing parameters with defaults or keep a separate `patchPilFields()` call.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Intent classification | New regex/NLP engine in PIL | `classify()` from `src/router/classifier/index.ts` | Already at <1ms, battle-tested in Phase 1, arch-tested for no-network constraint |
| Slash command dispatch | Manual string matching in optimize.ts | `registerSlash()` from `src/ui/slash/registry.ts` | Already wired into `dispatchSlash` in `app.tsx` |
| DB schema evolution | Dropping/recreating usage_events | `ALTER TABLE ... ADD COLUMN` in new migration | SQLite supports ADD COLUMN; existing data preserved |
| Timeout pattern | Custom Promise cancel logic | `Promise.race([fn(), new Promise(setTimeout)])` | Idiomatic Node/Bun — no library needed |
| Plain-text table rendering | ANSI table library | String padding with `.padEnd()` | D-17 explicitly calls out plain-text, headless-safe |

**Key insight:** PIL adds almost no new technology. It is a thin wiring layer over existing infrastructure. The implementation risk is in integration correctness (mutation point, store threading, schema migration), not algorithm complexity.

## Common Pitfalls

### Pitfall 1: SQLite STRICT + ALTER TABLE

**What goes wrong:** `ALTER TABLE usage_events ADD COLUMN pil_active INTEGER NOT NULL DEFAULT 0` fails on a STRICT table if `NOT NULL` is omitted and no DEFAULT is provided.
**Why it happens:** SQLite STRICT tables reject `NULL` in NOT NULL columns even for added columns if no DEFAULT is given.
**How to avoid:** Always include `DEFAULT 0` with `NOT NULL` for both new columns.
**Warning signs:** Migration throws on first run with existing DB. Write a migration test with an in-memory DB.

### Pitfall 2: recordUsageEvent Caller Contract Break

**What goes wrong:** Adding `pil_active` and `enrichment_delta` as required parameters to `recordUsageEvent()` breaks all existing call sites.
**Why it happens:** The function is called in multiple places (`orchestrator.ts`, tests).
**How to avoid:** Add new fields as optional trailing parameters with defaults (`pilActive = false, enrichmentDelta = 0`) OR add a separate `patchPilMetrics()` function that UPDATE the last-inserted row. The latter avoids touching the existing signature entirely.
**Warning signs:** TypeScript compilation errors at existing call sites.

### Pitfall 3: SlashContext.lastPrompt Write Path

**What goes wrong:** `SlashContext` is constructed per-dispatch in `app.tsx` and does not have a reference to the orchestrator's per-turn state.
**Why it happens:** The slash context is ephemeral — built at dispatch time, not shared with orchestrator.
**How to avoid:** Use the `pilStore` (module-level variable) pattern. `/optimize` reads `getPilLastResult()` directly — it does NOT need `SlashContext.lastPrompt` to be populated. `SlashContext.lastPrompt` can remain as an optional echo but is not the primary read path. D-15 says "populated with the enriched prompt" — this is satisfied by the store, not by orchestrator-to-slash direct coupling.
**Warning signs:** Trying to thread `SlashContext` through orchestrator's call chain.

### Pitfall 4: Promise.race Timeout with Stale Context

**What goes wrong:** The timeout branch resolves with an in-flight mutated `ctx` snapshot if the reference is captured after `runLayers()` starts.
**Why it happens:** JavaScript closures capture references, not values. If Layer 1 has already mutated `ctx` when the timeout fires, the "original" returned is actually partially-enriched.
**How to avoid:** Capture the original context as a snapshot (`const fallback = { ...ctx }`) before calling `runLayers()`, and resolve the timeout with `fallback`.
**Warning signs:** `/optimize` showing partial layer results on slow machines.

### Pitfall 5: Layer 1 Maps All Classifier "Hot" Results as Code Tasks

**What goes wrong:** The existing classifier returns `tier: 'hot'` for intents like `install` (package install), `run-command`, `search` — which are not "coding tasks" that need structured output optimization.
**Why it happens:** The classifier was built for routing decisions, not for PIL task-type discrimination.
**How to avoid:** Build a conservative mapping in Layer 1: only map reasons that clearly indicate code manipulation (`refactor`, `edit`, `create-file`, `tree-sitter:*`) to code `TaskType`s. Map `explain`, `search`, `install`, `run-command` to `analyze` or leave as `null` to avoid spurious optimization suffix injection.
**Warning signs:** Output optimization suffix appearing on `/route` and `/cost` command invocations.

### Pitfall 6: Arch Test Failure if PIL Imports EE

**What goes wrong:** Adding `import ... from '../ee/...'` in any file under `src/pil/` would fail `tests/arch/no-network-in-classifier.test.ts` only if the arch test scope is extended to cover `src/pil/`. However, if PIL is imported from Layer 3 (future), a broader arch test could catch it.
**Why it happens:** Layer 3 (EE Injection) is a stub now, so there is no EE import yet. But the pattern must be clean from day one.
**How to avoid:** Do NOT import EE client in `src/pil/` in this phase. Keep stubs completely empty (no imports). Add a comment `// TODO Phase 3: EE injection will import from src/ee/client.ts`.
**Warning signs:** Future arch test added for PIL catching EE imports in stubs.

### Pitfall 7: System Prompt Suffix Visibility in Headless/JSON Mode

**What goes wrong:** The suffix injected by Layer 6 may include multiline content that confuses headless mode JSON output parsing.
**Why it happens:** `--format json` headless mode wraps model output; extra whitespace in system prompt is harmless, but if the suffix accidentally injects a JSON escape sequence it could affect parsing.
**How to avoid:** Keep the suffix as clean ASCII strings. No JSON, no backticks that could conflict with markdown fences. Test `/optimize` in headless mode.
**Warning signs:** `tests/integration/headless-golden.test.ts` output diff failures.

   - What's unclear: Whether `app.tsx` reads from a module-level store or constructs `SlashContext` from individual sources (provider, model, cwd, lastPrompt).
   - Recommendation: Check `app.tsx`'s `dispatchSlash` call site. If `lastPrompt` is just passed from the last user input string, the orchestrator can populate a `pilLastPromptStore` module variable that `app.tsx` reads when building `SlashContext`. This is safer than threading a callback. The `/optimize` command reading `getPilLastResult()` directly from `src/pil/store.ts` is more reliable than `SlashContext.lastPrompt`.

2. **Should `enrichment_delta` be in tokens or bytes?**
   - What we know: D-12 says "estimated output token savings (naive: baseline_output_estimate − post-enrichment_output)". At the time of pipeline execution, no output tokens have been generated yet.
   - What's unclear: A true "savings" can only be computed after the model responds. Pre-send estimation is inherently speculative.
   - Recommendation: Store `enrichment_delta` as a signed integer representing estimated character reduction in the enriched prompt vs raw (enriched.length - raw.length). For output token savings, record actual delta at PostToolUse time using the `patchPilMetrics()` approach — update the last usage_events row with actual output tokens after the stream completes.

3. **Arch test coverage for src/pil/**
   - What we know: `tests/arch/no-network-in-classifier.test.ts` only covers `src/router/classifier/`. PIL has its own network-free hot-path requirement for Layer 1.
   - What's unclear: Whether a separate arch test for `src/pil/` is needed in this phase.
   - Recommendation: Add `tests/arch/no-network-in-pil-layer1.test.ts` covering `src/pil/layer1-intent.ts` with the same FORBIDDEN patterns check. This ensures Layer 1 stays network-free even when future developers add imports.
