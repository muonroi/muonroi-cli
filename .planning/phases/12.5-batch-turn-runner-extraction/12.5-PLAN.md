# Phase 12.5: Extract BatchTurnRunner from `orchestrator.processMessageBatchTurn`

> Decompose the batch-API streaming turn loop out of `orchestrator.ts` into a
> focused `BatchTurnRunner` class with callback DI. Mirrors the seam discipline
> used in Phase 12.1 (CouncilManager), 12.3 (StreamRunner), and 12.4
> (MessageProcessor).

---

## Motivation

After Phase 12.4 (MessageProcessor extracted, orchestrator.ts at 2848 LOC), the
next-biggest remaining concentration is `processMessageBatchTurn`
(lines 2185-2459, ~275 LOC). It owns the batch-API equivalent of the
MessageProcessor turn loop:

- Per-turn compaction (with overflow relax) before the request
- Batch chat-completions request build (assistant + tool-result messages,
  with optional sub-agent tool wiring, MCP tool merge, OAuth window pop)
- Polling loop for `pollBatchRequestResult`
- Tool execution loop with `executeBatchToolCall` (delegated back into the
  Agent)
- Provider-options shape capture (O1) for cost-leak forensics
- Observer notifications (`onStepStart` / `onStepFinish` / `onToolStart` /
  `onToolFinish` / `onError`)
- Transient retry with exponential backoff + jitter + harness `stream-retry`
  event emission
- Context-overflow recovery (single retry path with `relaxCompactionSettings`)
- MCP teardown via `closeMcp` in `finally`

Splitting it out isolates the batch-mode code path from the orchestrator god
class, brings the file under 2600 LOC, and aligns BatchTurnRunner's DI surface
with `MessageProcessorDeps` so a future hoist into a shared base interface is
mechanical.

## Naming choice

We keep the name **`BatchTurnRunner`** because:
- It mirrors the existing method name `processMessageBatchTurn`.
- "Runner" matches `StreamRunner` (Phase 12.3) — both own the lifecycle of a
  single streamed turn against a model endpoint.
- "Batch" disambiguates it from `MessageProcessor` (which owns the streaming
  path) and from "council turn" / "sprint turn" usages elsewhere.

`BatchTurnRunner.run(args)` is the canonical entrypoint.

## Delegation wiring choice

Two options were considered for wiring `MessageProcessor`'s
`deps.processMessageBatchTurn` callback after extraction:

1. **Keep the thin orchestrator wrapper.** `Agent.processMessageBatchTurn`
   becomes a one-line `yield* this.batchTurnRunner.run(args)` and
   MessageProcessorDeps continues to call `self.processMessageBatchTurn(...)`.
2. **Cut out the orchestrator wrapper.** Add a new
   `MessageProcessorDeps.getBatchTurnRunner: () => BatchTurnRunner` callback
   and have MessageProcessor call `runner.run(args)` directly.

**Chosen: Option 1 (thin wrapper).** Rationale: Option 1 keeps the
MessageProcessorDeps signature stable (no churn there) and matches the
existing precedent — `Agent.processMessage` is also a thin wrapper around
`MessageProcessor`. Option 2 would require expanding MessageProcessorDeps just
to save a one-line indirection, and would tightly couple MessageProcessor to
the BatchTurnRunner type. Option 1 is the smaller, lower-risk change.

The thin wrapper stays `private` on Agent so the existing
`processMessageBatchTurn: (args) => self.processMessageBatchTurn(args)`
callback in `_buildMessageProcessorDeps()` works unchanged.

| Transient-retry harness `stream-retry` event drops | Medium | Preserve the entire emit block verbatim, including the `(globalThis as Record<string, unknown>).__muonroiAgentRuntime` access pattern |
| MCP teardown skipped on early `return` paths | Medium | Keep the `finally { await closeMcp?.().catch(() => {}); }` block exactly as-is |
| Provider-options shape (O1) not captured per round | Low | The `setLastProviderOptionsShape` callback fires inside the inner-round loop — verify the call site is preserved |
| `executeBatchToolCall` cycle | Low | Expose via callback `executeBatchToolCall(tools, call, msgs, signal)` — same shape as the existing private method |
| Overflow-recovery loop `continue` semantics | Low | The body lives inside a `while (true)` — preserve as-is |
| Test suite regressions | Medium | Run full `bunx vitest run` after each commit; baseline 2688 pass |
