# Phase 12.3: Extract StreamRunner from `orchestrator.runTaskRequest`

> Decompose the sub-agent `streamText` driver out of `orchestrator.ts` into a focused `StreamRunner` class with DI.

---

## Motivation

After Phase 12.1 (CouncilManager extracted, 4605→~4923 LOC due to re-import overhead being smaller than expected and post-12.1 features) and Phase 12.2 (provider isolation), `orchestrator.ts` is still 4923 lines. The single largest remaining concentration of stream-handling logic is `runTaskRequest` (lines 1244–1714, ~470 lines) — the sub-agent driver. It owns:

- BashTool child wiring (verify sandbox prep + sandbox settings)
- Cumulative cap + read-budget + cross-turn-dedup tool wrapping
- Custom sub-agent resolution (explore / vision / verify / verify-detect / verify-manifest / computer / custom)
- Child model runtime resolution + capability gating
- MCP tool merging (agent-mode only)
- `streamText` configuration with `prepareStep` siliconflow-strip + subagent-compactor
- Full-stream consumption loop (text-delta, tool-call, error/finish dispatch under `MUONROI_DEBUG_SUBAGENT`)
- `onFinish` usage normalization (C1 cache split) + llm-done event emission
- Catch-block retry classification (re-throw transient errors when assistantText empty)
- G1 diagnostic stderr dumping for "No output generated"
- MCP teardown in `finally`

Splitting this out makes the sub-agent stream lifecycle independently testable, isolates the C1/G1/B3/F1 cost-leak code paths from the orchestrator god-class, and shrinks `orchestrator.ts` toward the ≤3500 line target.

> **Scope correction**: the original phase prompt estimated `runTaskRequest` at ~2000 lines and described a "tool invocation loop + write-ahead." Inspection shows the method is ~470 lines and is a single-streamText sub-agent driver — no tool invocation loop (that lives in `processMessage`) and no write-ahead (also in `processMessage`). The 3 plans below split along the natural seams in the actual code rather than the prompt's imagined structure.

## Plans

### Plan 12.3-01: Skeleton + StreamContext interface + setup extraction

**Goal:** Stand up `StreamRunner` with the DI contract and move sub-agent **setup** (child BashTool, child tools, MCP merge, runtime resolution, child prompt, child messages, debug-subagent header) into the class. `runTaskRequest` keeps the streamText body but constructs a `StreamRunner` for the setup phase.

Files:
- NEW `src/orchestrator/stream-runner.ts` — `StreamRunnerDeps` interface, `StreamRunner` class, `setup()` method returning a `PreparedSubAgentCall`
- NEW `src/orchestrator/__tests__/stream-runner-setup.test.ts`
- MODIFY `src/orchestrator/orchestrator.ts` — `runTaskRequest` calls `new StreamRunner(deps).setup(request, onActivity, signal)` and uses the returned object to invoke `streamText`

Commit: `refactor(orchestrator): 12.3-01 — extract StreamRunner skeleton + setup`

### Plan 12.3-02: Extract streamText invocation + prepareStep + onFinish + fullStream loop

**Goal:** Move the entire `streamText({...})` call site and `for await (const part of result.fullStream)` consumption loop into `StreamRunner.runStream()`. `prepareStep` and `onFinish` callbacks become private methods (`_prepareStep`, `_onFinish`).

Files:
- MODIFY `src/orchestrator/stream-runner.ts` — add `runStream()`, `_prepareStep()`, `_onFinish()`, `_handleStreamPart()`
- NEW `src/orchestrator/__tests__/stream-runner-handlers.test.ts`
- MODIFY `src/orchestrator/orchestrator.ts` — `runTaskRequest` body delegates to `streamRunner.runStream(prepared)`

Commit: `refactor(orchestrator): 12.3-02 — extract StreamRunner stream handlers`

### Plan 12.3-03: Extract error path + finalize `runTaskRequest` to thin shell

**Goal:** Move the catch block (transient retry re-throw, G1 diagnostic dump, failure ToolResult construction) and `finally` (MCP teardown) into `StreamRunner.run()`. `runTaskRequest` becomes ~30–40 lines: argv validation, build deps, `new StreamRunner(deps).run(input)`, return.

Files:
- MODIFY `src/orchestrator/stream-runner.ts` — add `run()` orchestrating `setup() + runStream() + error handling`
- NEW `src/orchestrator/__tests__/stream-runner-error-paths.test.ts`
- MODIFY `src/orchestrator/orchestrator.ts` — `runTaskRequest` reduces to a thin shell

Commit: `refactor(orchestrator): 12.3-03 — finalize StreamRunner, thin runTaskRequest`

## Risk Assessment

| Risk | Level | Mitigation |
|---|---|---|
| Cost-leak regression (F1/G1/B3/C1) | High | Run cost-leak harness specs after every plan |
| `_currentCallId` race when stream throws before onFinish | Low | Mirror existing behavior — set in setup, reset in onFinish/catch |
| `_lastProviderOptionsShape` shape capture timing | Low | Capture happens exactly once before streamText; preserve order |
| MCP teardown in `finally` vs throw in setup | Low | `closeMcp` captured in outer scope before try, finally guard with `?.` |
| Verify-sandbox prep is async and runs BEFORE try/finally | Low | Keep the prep call OUTSIDE `run()` or inside `setup()` consistently |
| StreamRunner allocation per call | Negligible | Sub-agent path is already heavyweight (BashTool spawn + MCP probe) |
