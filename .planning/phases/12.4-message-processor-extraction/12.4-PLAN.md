# Phase 12.4: Extract MessageProcessor from `orchestrator.processMessage`

> Decompose the main streaming turn loop out of `orchestrator.ts` into a focused
> `MessageProcessor` class with DI. Mirrors the seam discipline used in Phase
> 12.1 (CouncilManager) and Phase 12.3 (StreamRunner).

---

## Motivation

After Phase 12.3 (StreamRunner extracted, orchestrator.ts at 4447 LOC), the
single largest remaining concentration in `orchestrator.ts` is
`processMessage` (lines 2580–4289, ~1709 LOC). It owns:

- PIL enrichment (Pipeline layers 1/3/6 — invocation, fail-open, logging)
- Per-turn ROUTE-11 routing decision (decide → routedModel + fallback to
  non-disabled provider via CouncilManager helpers)
- Vision proxy (history + current turn, fail-open)
- System prompt assembly (chitchat gate, playwright guidance, PIL suffix)
- Auto-council gate (PIL taskType + heavy tier + role count)
- Tool roundtrip loop (top-level `streamText` with prepareStep + onFinish +
  fullStream consumer), including SAMR phase1/phase2 model switch
- Write-ahead persistence (Phase A4 tool_calls, A5 message_seq)
- Cross-turn dedup (C3) wrap
- Top-level cumulative cap (F1) + read budget + dedup wrap
- ProviderOptions composition (F1 promptCacheKey via buildTurnProviderOptions,
  G1 unsupportedParams drop via shouldDropParam, anthropic thinking budget
  override by PIL taskType, O1 providerOptions shape capture)
- Top-level compaction (B4)
- Cost recording (per step + final)
- Transient retry loop with exponential backoff
- Context-overflow recovery loop
- Hook firing (SessionStart, UserPromptSubmit, PreToolUse, PostToolUse,
  PostToolUseFailure, Stop, StopFailure)
- Observer notifications (onStepStart/Finish, onToolStart/Finish, onError)
- Debug trace recording (pipeline summary)
- ROUTE-11 feedback (success/cancelled/fail)

Splitting this out makes the streaming turn loop independently testable,
isolates the C3/F1/B4 cost-leak code paths from the orchestrator god-class,
and shrinks `orchestrator.ts` toward the ≤3500 line target.

## Naming choice

We keep the name **`MessageProcessor`** — it mirrors the public method name
(`processMessage`) and matches the precedent set by `CouncilManager` /
`StreamRunner`. `TurnProcessor` was considered but rejected because "turn"
already overlaps with the batch-turn helper `processMessageBatchTurn` and the
council "turn" concept. `MessageProcessor.run()` is the canonical entrypoint.

## Plans

## Risk Assessment

| Risk | Level | Mitigation |
|---|---|---|
| Cost-leak regression (F1/B4/C3) | High | Run cost-leak harness specs after every plan |
| `this.messages` mutation semantics differ via getter+setter | Medium | Use `getMessages()` / `setMessages()` callbacks that read/write the Agent's array reference directly; `pushMessage()` / `pushMessageSeq()` for appends |
| `_currentCallId` race across `onFinish` and stream loop | Low | Mirror existing behavior — keep the local `_topCallId` const inside `MessageProcessor.run()` so the closure capture is identical |
| `onFinish` closing `_currentCallId` after the runtime is gone | Low | Setter callback handles it; the `try/catch` block remains identical |
| Auto-council recursion through `processMessage` itself | Low | The recursive call goes through the public `Agent.processMessage` (thin wrapper) which constructs a new MessageProcessor — no recursion inside the class |
| Test suite regressions | Medium | Run full `bunx vitest run` after each commit; baseline against Phase 12.3 numbers |
