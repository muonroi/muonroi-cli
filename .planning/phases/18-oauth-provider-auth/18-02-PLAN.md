<objective>
Add Google Gemini OAuth using browser-redirect (Authorization Code + PKCE) flow.
OpenAI Device-Code path stays unchanged. Both providers coexist; token store and
ProviderOAuth interface require zero modification.

Acceptance: `keys login google` opens browser to Google authorize URL, loopback
callback captures code, exchanges for tokens, persists. With tokens stored,
`-m gemini-1.5-flash` works without GOOGLE_API_KEY. Refresh and revoke work.
</objective>

## Open questions ŌĆö decisive answers

1. **client_id**: Use gemini-cli's published OAuth client credentials from
   `packages/core/src/code_assist/oauth2.ts` (MIT-licensed OSS):
   Source: https://github.com/google-gemini/gemini-cli
   Rationale: same reasoning as OpenAI client_id in 18-01 ŌĆö OSS MIT-licensed,
   publicly committed in a major Google-owned repo, zero registration friction.
   Not secret: embedded in a public JavaScript bundle, standard for native/CLI apps
   (RFC 8252). Values stored in code; override via env var MUONROI_GOOGLE_CLIENT_SECRET.

2. **Flow**: Authorization Code + PKCE with loopback redirect (not device-code).
   Google does not support device-code for general OAuth 2.0 anymore for new clients.
   Reuse `src/mcp/oauth-callback.ts` via a thin wrapper ŌĆö no extraction needed
   since the existing helper is already generic enough.

3. **Scopes**: `https://www.googleapis.com/auth/cloud-platform openid email`
   ŌĆö same as gemini-cli uses. cloud-platform covers Generative Language API.

4. **Token endpoint**: `https://oauth2.googleapis.com/token`
   **Auth endpoint**: `https://accounts.google.com/o/oauth2/v2/auth`
   **Revoke endpoint**: `https://oauth2.googleapis.com/revoke`
   **Userinfo endpoint**: `https://openidconnect.googleapis.com/v1/userinfo`

5. **authHeaders**: Standard `{ Authorization: "Bearer {access_token}" }` ŌĆö
   no special account header (unlike OpenAI).

6. **loopback-callback.ts**: NOT extracting ŌĆö `src/mcp/oauth-callback.ts` is already
   clean and reusable. Import directly.

7. **browser-flow.ts**: New file with generic browser-redirect helpers:
   `buildAuthorizeUrl()`, `exchangeBrowserCode()`. Separate from `device-flow.ts`
   which is device-code only.

## Test strategy ŌĆö mock approach for browser-redirect

The `startOAuthCallbackServer` is injected via a constructor parameter `callbackServerFn`
on GeminiOAuthProvider (defaults to the real one). Tests supply a mock that immediately
fires `onCode("test-auth-code", "test-state")`. The `openBrowser` function is similarly
injectable (default: `child_process.exec`). This keeps tests purely synchronous
with no real HTTP servers or browser opens.

All token endpoint calls go through injectable `fetchFn` (same pattern as OpenAI).
