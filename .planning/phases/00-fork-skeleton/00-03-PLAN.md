| user-machine ↔ filesystem | Storage path defines where API keys, sessions, transcripts land on disk. A wrong path could collide with grok-cli on the same box (PII cross-pollination) or land in an unexpected home subdirectory. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|---|---|---|---|---|
| T-00.03-01 | Information Disclosure | Storage namespace collision with grok-cli on same machine | mitigate | `.muonroi-cli/` (D-002) is distinct from `.grok/`. Verified by grep — zero residual `.grok` references. |
| T-00.03-02 | Tampering | Test fixtures retaining stale `.grok/` paths could mask regressions | mitigate | Vitest run regenerates snapshots; commit includes the regenerated artifacts. |
| T-00.03-03 | Repudiation | Migration of existing `~/.grok/` sessions could mask whether a user's data is muonroi-cli's or grok-cli's | accept | Clean break per `Out of Scope > Migrating existing ~/.grok/ sessions`. Documented in commit message. |
| T-00.03-04 | Elevation of Privilege | A misnamed credentials path could expose keys to other tools' read-permissions | mitigate | Single namespace under user's home directory, OS perms inherited from grok-cli's existing pattern. Keytar-based key load (not file-based) lands in plan 00.05. |
</threat_model>
