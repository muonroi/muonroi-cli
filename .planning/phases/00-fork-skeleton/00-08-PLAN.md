windows-smoke.yml shape (per 00-CONTEXT.md "CI in Phase 0"):

```yaml
name: windows-smoke
on:
  push:
    branches: [master]
  pull_request:
    branches: [master]
permissions:
  contents: read
jobs:
  smoke:
    runs-on: windows-latest
    timeout-minutes: 15
    steps:
      - uses: actions/checkout@v4
      - uses: oven-sh/setup-bun@v2
        with:
          bun-version: ">=1.3.13"
      - name: Install
        run: bun install --frozen-lockfile
      - name: Typecheck
        run: bunx tsc --noEmit
      - name: Test
        run: bunx vitest run
      - name: Boot smoke (headless)
        shell: pwsh
        run: |
          # Start the TUI in the background, give it 3s to render, then send SIGINT (Ctrl+C).
          # On Windows, use Stop-Process with -Force as a SIGINT proxy if no graceful Ctrl+C signal is reachable.
          # We assert: process started, lived >= 1s (proves boot succeeded), exited within 3s of signal.
          $proc = Start-Process bun -ArgumentList "run","src/index.ts","--prompt","__SMOKE__" -PassThru -NoNewWindow
          Start-Sleep -Seconds 3
          if (-not $proc.HasExited) {
            $proc.CloseMainWindow() | Out-Null
            Start-Sleep -Seconds 2
            if (-not $proc.HasExited) { Stop-Process -Id $proc.Id -Force }
          }
          if ($proc.ExitCode -ne 0 -and $proc.ExitCode -ne 130) {
            Write-Error "Boot smoke failed with exit code $($proc.ExitCode)"
            exit 1
          }
```

NOTE: the boot smoke is intentionally minimal — exits-cleanly is the only assertion. A "renders within 3s" assertion via node-pty / OpenTUI capture is deferred to Phase 1 per 00-CONTEXT.md "Automated TUI render tests via node-pty — Phase 1 once we have the harness need."

NOTE: `--prompt "__SMOKE__"` runs the headless one-shot path (no interactive shell). The Anthropic provider will fail on this prompt because there's no API key in CI — that's OK; the smoke is testing BOOT, not Anthropic. The provider-load failure is caught at main()'s catch, redacted, and the process exits with code 1. To distinguish "boot worked, no key" from "boot broke", we need a smoke flag that boots-and-exits without invoking the provider. Add this:

  - In src/index.ts main(), if `process.argv` contains `--smoke-boot-only`, run loadConfig + loadUsage + (skip key load) + mountTUI(skipModelInit=true) + immediately exit 0. This is a Phase 0 testing affordance only.

Updated workflow step:

```yaml
      - name: Boot smoke (headless)
        shell: pwsh
        run: |
          $proc = Start-Process bun -ArgumentList "run","src/index.ts","--smoke-boot-only" -PassThru -NoNewWindow -Wait
          if ($proc.ExitCode -ne 0) {
            Write-Error "Boot smoke failed with exit code $($proc.ExitCode)"
            exit 1
          }
```

## Trust Boundaries

| Boundary | Description |
|---|---|
| GitHub Actions runner ↔ npm registry | CI install pulls from npm — frozen-lockfile prevents drift. |
| GitHub Actions runner ↔ external services | Boot smoke does NOT contact Anthropic / EE / Qdrant — `--smoke-boot-only` short-circuits before any of them. |
| Repo ↔ secrets | No secrets needed in Phase 0 CI (no Anthropic key, no EE token). Phase 4 will add them via Actions secrets — deferred. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|---|---|---|---|---|
