# Phase 21 — Plan 02: Toast UI + `/ee-context` + Timeout Overrides + Harness Spec

**Status:** Not started — DEPENDS ON Plan 01 (logger + protocol + event refactor must land first).
**Phase:** 21 EE Observability & Resilience
**Estimated effort:** 1 session (~45-60 min)

## Goal

Make the structured EE signals from Plan 01 visible to users and verifiable in CI:
1. Render a passive toast when BB retrieval times out.
2. Expose the `eeBBContext` setting via a slash command.
3. Let operators override timeouts via env.
4. Lock the contract with a harness E2E spec.

## Prerequisites from Plan 01

- `src/utils/ee-logger.ts` exists with `logEeFailure(source, kind, err, extra)`.
- Protocol exports `ee-timeout` and `ee-error` event kinds.
- 12 silent EE catches now emit events with stable `source` strings (especially `bridge.searchByText` and `bb-retrieval.<...>` — Plan 01 covers the bridge layer; this plan also covers `bb-retrieval.ts` itself if any silent catch remains).

## Tasks

### T1 — Toast React component
`src/ui/components/Toast.tsx`:

- Wrap render root with `<Semantic id="toast" role="toast" name={text}>`.

- Use existing styling primitives from `src/ui/components/` (do not introduce new styling deps).

### T2 — Toast subscriber in app
- In `src/ui/app.tsx`, hook into the same agent-runtime sink used by `emitEvent` and listen for `kind === "toast"`.
- Render the most recent toast (max 1 visible). Auto-dismiss on timer or new toast.
- When an `ee-timeout` with `source: "bb-retrieval.*"` arrives, push a one-shot toast `running without BB context (EE slow or down)`. Debounce: once per session ID via `useRef`.

### T3 — `/ee-context` slash command
Add command at the same dispatcher that handles `/config` (`src/ui/app.tsx:3692-3701`):
- `/ee-context on` → set `userSettings.eeBBContext = true`, save, push toast `BB context: ON`.
- `/ee-context off` → set to false, push toast `BB context: OFF`.
- `/ee-context status` → push toast `BB context: <current>`.
Update help/slash-menu listing.

### T4 — Env overrides for timeout constants
- `src/pil/layer3-ee-injection.ts:30` — replace literal 1500 with helper:
  ```ts
  function readTimeoutEnv(name: string, def: number, lo: number, hi: number): number {
    const raw = Number(process.env[name]); return Number.isFinite(raw) ? Math.max(lo, Math.min(hi, raw)) : def;
  }
  const PIL_SEARCH_TIMEOUT_MS = readTimeoutEnv("MUONROI_PIL_SEARCH_TIMEOUT_MS", 1500, 500, 5000);
  ```
- `src/ee/bb-retrieval.ts:85` — same treatment with `MUONROI_BB_RETRIEVAL_TIMEOUT_MS`, default 800, clamp `[300, 3000]`.
- Re-export `readTimeoutEnv` from `src/utils/ee-logger.ts` (single place; both timeout files import from there).

### T6 — Harness E2E spec `tests/harness/ee-timeout.spec.ts`
Use the `spawnHarness()` helper. Layout:
- Boot TUI with `--agent-mode --mock-llm <fixture>`.
- Set `MUONROI_BB_RETRIEVAL_TIMEOUT_MS=100` in the spawn env to force timeout.
- (Optional) point BB retrieval at a stub URL or rely on the natural timeout against unreachable EE.
- Send a normal user prompt via `driver.type` + `driver.press("Enter")`.
- Assert via `driver.events()`:
  - An `ee-timeout` event with `source` starting with `bb-retrieval` arrives within 1500ms.
- Assert via `driver.wait_for({ selector: 'id=toast', timeoutMs: 2500 })`:
  - Toast appears; `driver.query('id=toast')?.name` includes `without BB context`.
- Assert the prompt itself produces an assistant message — degrade contract holds.

## Risks

- `useRef` debounce must reset across logical sessions (new spawnHarness boot = new session id). Verify the session-id source before keying.
- `driver.events()` replay (CLAUDE.md:184) ensures late-subscribe still sees the event; no need to subscribe before dispatch.
- If `bb-retrieval` path has its own internal Promise.all wrapper that catches before the timeout reaches the outer catch, the event may emit twice. Verify after Plan 01 lands.
