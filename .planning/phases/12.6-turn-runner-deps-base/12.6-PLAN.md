# Phase 12.6 — Hoist `TurnRunnerDepsBase`

> Pure type-level refactor. Zero runtime change. Extracts the overlap between
> `MessageProcessorDeps` (12.4) and `BatchTurnRunnerDeps` (12.5) into a shared
> base interface so both surfaces declare the relationship explicitly via
> `interface X extends TurnRunnerDepsBase`.

## Audit — overlap between the two deps interfaces

Properties whose **name AND signature are identical** in both interfaces:

### State references
- `readonly messages: ModelMessage[]`
- `readonly bash: BashTool`
- `readonly mode: AgentMode`
- `readonly maxToolRounds: number`
- `readonly schedules: import("../tools/schedule").ScheduleManager`
- `readonly sendTelegramFile: ((filePath: string) => Promise<ToolResult>) | null`

### Scalar getters/setters
- `getCompactedThisTurn(): boolean`
- `setCompactedThisTurn(v: boolean): void`
- `setLastProviderOptionsShape(shape: string | null): void`

### Compaction behavior delegators
- `getCompactionSettings(contextWindow?: number): CompactionSettings`
- `compactForContext(provider, system, contextWindow, signal, settings?, overflow?): Promise<boolean>`
- `postTurnCompact(provider, system, contextWindow, signal): Promise<void>`

### Task / delegation tools
- `runTask(request: TaskRequest, signal?: AbortSignal): Promise<ToolResult>`
- `runDelegation(request: TaskRequest, signal?: AbortSignal): Promise<ToolResult>`
- `readDelegation(id: string): Promise<ToolResult>`
- `listDelegations(): Promise<ToolResult>`

### Turn bookkeeping
- `appendCompletedTurn(userMessage: ModelMessage, assistantMessages: ModelMessage[]): void`
- `discardAbortedTurn(userMessage: ModelMessage): void`
- `recordUsage(usage: { totalTokens?, inputTokens?, outputTokens?, cacheReadTokens?, cacheCreationTokens? } | undefined, source?: UsageSource, model?: string): void`

### Near-overlaps (LEFT ALONE — different shape or only in one)
- `MessageProcessorDeps` carries many extra members the streaming path needs
  (PIL state, hook firing, council manager, abort context, write-ahead seqs,
  etc.) — not present in batch path, so they stay where they are.
- `BatchTurnRunnerDeps` has `maxTokens`, `getSessionId`, `getBatchClientOptions`,
  `createTools`, `executeBatchToolCall` — unique to the batch API path, stay
  in `BatchTurnRunnerDeps`.
- No near-overlap collisions found (no property has the same name with
  different signature across the two interfaces).
