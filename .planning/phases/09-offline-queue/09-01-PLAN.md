  <behavior>
    RED tests (write these FIRST, all must fail):

    1. enqueue() creates queue directory lazily (D-11):
       - Call enqueue() with homeOverride=tmpDir
       - Expect directory `{tmpDir}/.muonroi-cli/ee-offline-queue/` to exist
       - Expect exactly 1 JSON file matching /^\d+-[a-z0-9]{4}\.json$/

    2. enqueue() writes correct JSON content (D-03):
       - Enqueue entry { endpoint: "/api/feedback", body: { test: 1 }, enqueuedAt: 1000 }
       - Read the written file, parse JSON
       - Expect parsed object to deep-equal the input entry

    3. enqueue() enforces 100-entry cap by deleting oldest (D-04):
       - Pre-populate queue dir with 100 files (timestamp-named)
       - Call enqueue() with a new entry
       - Expect directory has exactly 100 files (not 101)
       - Expect the oldest pre-populated file is gone
       - Expect the new file exists

    4. getQueueDir() returns correct path:
       - getQueueDir() returns `{os.homedir()}/.muonroi-cli/ee-offline-queue`
       - getQueueDir("/tmp/test") returns `/tmp/test/.muonroi-cli/ee-offline-queue`

    5. drainQueue() replays entries in FIFO order:
       - Pre-populate 3 entries with timestamps 1000, 2000, 3000
       - Start a stub HTTP server that records request bodies
       - Call drainQueue() (awaitable version for test) with that server
       - Expect 3 requests received in order (1000, 2000, 3000 by enqueuedAt)
       - Expect all 3 files deleted from queue dir

    6. drainQueue() stops on failure, leaves remaining entries (D-07):
       - Pre-populate 3 entries
       - Stub server returns 500 on 2nd request
       - Call drainQueue()
       - Expect 1st file deleted (success), 2nd and 3rd files remain

    7. drainQueue() handles non-existent queue directory:
       - Call drainQueue() without creating any queue dir
       - Expect no error thrown, function returns cleanly

    8. drainQueue() discards corrupt JSON files:
       - Write a file with invalid JSON content to queue dir
       - Write a valid queue entry after it
       - Call drainQueue()
       - Expect corrupt file deleted, valid file replayed and deleted

    9. Filename format matches D-02:
       - Call enqueue()
       - Verify filename matches /^\d+-[a-z0-9]{4}\.json$/

    10. drainQueue() is fire-and-forget when called via void wrapper (D-08):
        - Verify drainQueue returns void (not Promise) from the fire-and-forget wrapper
        - The internal async IIFE runs without being awaited by caller
  </behavior>

  <implementation>
    After all RED tests are written and confirmed failing, implement src/ee/offline-queue.ts:

    1. Constants: MAX_QUEUE_SIZE = 100, QUEUE_DIR_NAME = "ee-offline-queue"

    2. getQueueDir(homeOverride?: string): string
       - Returns path.join(homeOverride ?? os.homedir(), ".muonroi-cli", QUEUE_DIR_NAME)
       - Per D-01: uses ~/.muonroi-cli/ee-offline-queue/

    3. QueueEntry interface (inline, not exported to types.ts — keep module self-contained):
       ```typescript
       interface QueueEntry {
         endpoint: string;    // "/api/feedback", "/api/extract", "/api/prompt-stale"
         body: unknown;       // original request payload
         enqueuedAt: number;  // Date.now() at enqueue time
       }
       ```

    4. makeFilename(): string
       - Per D-02: `${Date.now()}-${Math.random().toString(36).slice(2, 6)}.json`

    5. getSortedFiles(dir: string): Promise<string[]>
       - readdir, filter .json, sort lexicographically (natural FIFO per timestamp prefix)

    6. enqueue(entry: QueueEntry, homeOverride?: string): Promise<void>
       - Per D-11: await fs.mkdir(dir, { recursive: true }) — lazy init
       - Per D-04: if files.length >= 100, unlink oldest (files[0])
       - Write JSON.stringify(entry) to new file with makeFilename()

    7. drainQueue(fetchImpl: typeof fetch, headers: Record<string, string>, baseUrl: string, homeOverride?: string): void
       - Per D-08: fire-and-forget wrapper — void (async () => { ... })()
       - Per D-06: sequential loop over getSortedFiles()
       - For each file: read, parse JSON, POST to baseUrl + entry.endpoint
       - On success (resp.ok): unlink file, continue
       - On parse error: unlink corrupt file, continue
       - Per D-07: on HTTP error or network failure: break loop (leave file + remaining)
       - On ENOENT from readdir: return early (D-11 lazy init means dir may not exist)
       - Replay timeout: AbortSignal.timeout(5000) per research pattern

    8. Export drainQueueAsync for testing (returns Promise instead of void):
       - Same logic but returns the promise so tests can await completion

    All imports: node:fs/promises, node:os, node:path, node:crypto (or Math.random)
    NO imports from client.ts or intercept.ts (prevents circular deps per research pitfall 3)
