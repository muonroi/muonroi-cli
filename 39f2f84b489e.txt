## Phân tích kiến trúc tối giản cho Extension PDF Text Selection Translator
Với vai trò **Kiến trúc sư Extension**, tôi tập trung vào việc thiết kế một hệ thống **tối giản, khả thi**, có thể hoạt động ổn định trên PDF viewer của Chrome và Brave (dựa trên Chromium, dùng PDF.js hoặc built-in viewer). Dưới đây là các quyết định kiến trúc chính, dựa trên dữ liệu nghiên cứu hiện có.
### 1. Lựa chọn Manifest Version: **Bắt buộc Manifest V3**
- **Bằng chứng:** Google đã ngừng hỗ trợ Manifest V2 từ đầu 2024, Brave cũng theo lộ trình tương tự. Mọi extension mới phải dùng V3.
- **Hệ quả:** Background page chuyển thành **Service Worker** (chỉ chạy khi cần, không có DOM). Cần cấu trúc lại luồng xử lý để tránh mất state (ví dụ: sử dụng storage API để lưu tạm text cần dịch trước khi gọi fetch).
### 2. Cách bắt văn bản được chọn trong PDF viewer
**Giải pháp tối giản: `chrome.contextMenus` API với context `"selection"`**
- **Bằng chứng:** Theo nghiên cứu, `chrome.contextMenus` hoạt động tốt trên PDF viewer của Chrome và Brave `[nguồn: StackOverflow]`. Khi người dùng chọn text và click chuột phải, background script nhận được `info.selectionText` **mà không cần inject content script vào trang PDF**.
- **Đây là kiến trúc tối ưu cho phiên bản đầu:** Không cần xử lý phức tạp `web_accessible_resources`, không cần inject script vào `chrome-extension://` pages. Giảm thiểu rủi ro về bảo mật và tương thích.
**Tuy nhiên, cần lưu ý hai kịch bản phụ:**
- Nếu muốn hiển thị **tooltip trực tiếp bên cạnh vùng chọn** (hover popup), thì bắt buộc phải inject code vào PDF viewer thông qua kỹ thuật `document.createElement('script')` + `web_accessible_resources`. Nhưng đây là tính năng **nâng cao**, có thể bỏ qua ở phiên bản tối giản.
- Người dùng có thể muốn dịch text từ **trang web thông thường (HTML)** – khi đó content script sẽ inject được bình thường, và có thể dùng `window.getSelection()` để lấy text.
**Khuyến nghị kiến trúc:**
Cả hai luồng đều đưa về background service worker để xử lý dịch.
### 3. Luồng xử lý và message passing
Sơ đồ luồng tối giản:
[User chọn text trong PDF]
        │
        ▼
[Context Menu click] → background.js (service worker)
        │
        ▼
[Gọi translation API (Google/DeepL/...)]
        │
        ▼
[Hiển thị kết quả: popup hoặc notifications]
- **Background service worker** chịu trách nhiệm:
  - Lắng nghe `chrome.contextMenus.onClicked` (cho PDF)
  - Lắng nghe message từ content script (cho web page)
  - Gọi API dịch thuật (fetch)
  - Lưu lịch sử hoặc cache qua `chrome.storage.local`
- **Content script** (dành cho web page) khi nhận selection sẽ gửi message `{action: "translate", text: selection}` đến background.
- **Popup** (khi click icon extension) có thể hiển thị kết quả dịch gần nhất, hoặc cho phép người dùng nhập tay.
**Message passing đơn giản:**
// content.js (chỉ dùng cho web page)
chrome.runtime.sendMessage({ action: "translate", text: selectedText });
Background SW nhận message và gọi API, sau đó có thể gửi kết quả lại qua `chrome.runtime.onMessage` nếu cần (hoặc dùng `chrome.notifications` để hiển thị).
### 4. Tương thích với PDF.js vs built-in viewer
- Cả Chrome và Brave đều dùng **built-in PDF viewer** (một extension ẩn với ID `mhjfbmdgcfjbbpaeojofohoefgiehjai`). Viewer này dựa trên PDF.js, nhưng nó chạy trong một `chrome-extension://` page.
- **Không có sự khác biệt về mặt kiến trúc** giữa “PDF.js viewer” và “built-in viewer” đối với extension bên ngoài – cả hai đều là `chrome-extension://` page, không thể inject content script trực tiếp.
- Giải pháp `chrome.contextMenus` hoạt động giống hệt nhau trên cả hai nền tảng.
- Nếu Brave có tùy chỉnh riêng (ví dụ: dùng PDF.js mới hơn), thì context menu vẫn hoạt động vì nó là API của Chrome extension, không phụ thuộc vào nội dung trang.

### Chuyên viên Tương tác PDF (`research` · deepseek-v4-flash)
## Analysis from the PDF Interaction Lens
As **Chuyên viên Tương tác PDF**, I focus strictly on **how to reliably extract selected text** from PDF viewers in Chrome/Brave. The core challenge is that PDF viewers run in restricted environments (e.g., `chrome-extension://` URLs, shadow DOMs), making standard content script injection unreliable. Here’s my evidence-based assessment:
### 1. Text Selection Events & Reliable Capture
- **Primary mechanism:** `chrome.contextMenus` with `contexts:["selection"]` – the most reliable cross-viewer method. It captures the user’s selection text **without needing to inject scripts** into the PDF viewer’s context. This works on both Chrome’s built-in PDF viewer (using PDFium) and Brave’s viewer (identical engine), as confirmed by [StackOverflow community](https://stackoverflow.com/questions/15527095/how-extension-get-the-text-selected-in-chrome-pdf-viewer) and our research findings.
- **Why it works:** The browser’s selection event is handled at the native level, not within the page’s JavaScript. The `selectionText` from the context menu API returns the plain text of the current selection, regardless of shadow DOM encapsulation or extension scheme origin.
- **Limitation:** Requires a right-click action. For a seamless UX (e.g., floating button after selection), we would need alternative capture, but for MVP this is acceptable.
### 2. Accessibility APIs (Alternative Path)
- PDF viewers typically expose an accessibility tree (e.g., Chrome’s PDF viewer uses `chrome.accessibilityFeatures` or `aria-*` attributes). However, programmatically extracting the **current selection** from the accessibility tree is **not standardized** and varies across versions. It’s fragile and **not recommended** for a first release.
- **Fallback use:** If contextMenus fails in a non-standard PDF viewer, we could attempt to read the `aria-selected` or `document.activeElement` from an injected script (see next point), but this is low priority.
### 3. Shadow DOM Challenges
- Chrome’s built-in PDF viewer uses a shadow root within an embed element. Direct `content_scripts` cannot access it because the URL scheme `chrome-extension://` blocks injection.
- **Solution:** Inject a `<script>` tag from a content script running on `<all_urls>` (except the PDF viewer page) using `web_accessible_resources`. This script can then traverse the shadow DOM via `document.querySelector('embed').shadowRoot` and attach selection listeners. However, this is **complex and brittle** – the shadow DOM structure can change with Chrome updates.
- **Recommendation:** Avoid this for MVP. Rely on `chrome.contextMenus` as the primary path; reserve injection for advanced features (e.g., live translation tooltip).
### 4. Fallback Strategies
- **Best fallback:** Combine `contextMenus` with a user-configurable keyboard shortcut (e.g., `Ctrl+Shift+T`) that triggers the same background handler – but the shortcut would need focus on the PDF viewer (which may work if the PDF tab is active).
### 5. Specific Challenges for Built-in Viewer vs. PDF.js
- **Chrome/Brave built-in viewer:** Uses native PDFium – selection events are standard. `contextMenus` works perfectly.
- **PDF.js (e.g., Firefox, or some websites):** The text layer is a normal DOM element (divs with `textLayer` class). Content scripts **can** inject and `window.getSelection()` works. Extension can attach mouseup/selectionchange events directly. However, note that PDF.js on webpages (not extension pages) is allowed.
- **For a cross-browser approach:** Detect which PDF viewer is being used via URL or DOM check; use `contextMenus` for native viewers, content script injection for PDF.js.
### 6. Security & Privacy in Extraction
- No user data is sent to third parties during extraction – the selection text is only sent to the translation API after user action (context menu click). This is safe.
- If we later add automatic detection, we must ensure selection text is not leaked without consent.

## Conclusion for the Extension Architect
Based on the above, I propose these **architectural constraints** for the first version:
1. **Primary extraction method:** `chrome.contextMenus` with `contexts:["selection"]` – no need to inject scripts into PDF viewers.
2. **Backup method** (for PDF.js on webpages): A content script that listens to `mouseup` and sends `window.getSelection().toString()` via `chrome.runtime.sendMessage`. This script runs on `<all_urls>` but must be careful to avoid interfering with PDF viewer embed pages (we can check for `embed[type="application/pdf"]` as a filter).
3. **No Shadow DOM traversal** in v1 – too fragile.
4. **Future enhancement:** Add a floating button that appears when text is selected, which requires injecting into the PDF viewer’s context via `web_accessible_resources` – but this is deferred.

Specifically, should we use a single `contextMenus` listener for all selections, or create separate logic for when the content script can also capture selection?

**Microsoft Translator là lựa chọn cân bằng nhất** cho extension miễn phí, vì:
1. **Free tier hào phóng nhất** — 2M ký tự/tháng gấp 4 lần Google/DeepL. Với một extension dịch text selection, người dùng ít khi chạm trần này.
2. **179 ngôn ngữ** — phủ toàn bộ tiếng Việt, ngôn ngữ châu Á, và các ngôn ngữ ít phổ biến mà DeepL không có.

4. **Chi phí thấp nhất nếu vượt hạn mức** — $10/triệu ký tự, quan trọng nếu sau này bạn muốn có gói trả phí.
**Với điều kiện:** Extension nên để người dùng **tự cấp API key** (BYOK — Bring Your Own Key) thay vì nhúng key cứng. Điều này:
- Tránh rủi ro lộ key trong mã nguồn extension.
- Mỗi người dùng dùng quota riêng, không bị giới hạn tập trung.

**Nếu chất lượng dịch là ưu tiên tuyệt đối** (đặc biệt cho người dùng châu Âu), DeepL là vua — nhưng giới hạn ~30 ngôn ngữ là điểm yếu lớn cho extension toàn cầu. Có thể cân nhắc hỗ trợ **multi-provider**: ưu tiên DeepL nếu ngôn ngữ được hỗ trợ, fallback sang Microsoft Translator.
**LibreTranslate** hấp dẫn về mặt bảo mật (dữ liệu không rời máy), nhưng chất lượng dịch thấp hơn hẳn và yêu cầu người dùng phổ thông tự host một Docker container là **rào cản quá lớn**. Tôi không khuyến nghị cho MVP.

### 1. Điểm đồng thuận
Tôi hoàn toàn nhất trí với chiến lược trích xuất văn bản của bạn:
- **`chrome.contextMenus` với `contexts:["selection"]` là lựa chọn tối ưu cho MVP.** Từ góc nhìn API, đây là phương án **đơn giản hóa đáng kể** việc tích hợp: background script nhận `selectionText`, gọi API dịch, trả kết quả – không cần đụng đến shadow DOM hay injection phức tạp. Mỗi phương án trích xuất bạn liệt kê đều trả về plain text, và plain text chính là thứ mọi API dịch thuật đều nhận – nên về mặt dữ liệu, không có khác biệt.
- **Tách biệt logic native-PDF vs PDF.js là đúng đắn.** PDF.js chạy trên trang web thông thường, nơi content script có thể inject và dùng `window.getSelection()` – đồng nghĩa ta có thể hiển thị tooltip dịch trực tiếp (inline). Còn native PDF viewer thì bắt buộc dùng context menu → hiển thị popup/notification. Điều này ảnh hưởng đến UX dịch thuật, không chỉ extraction.

### 2. Điểm bổ sung quan trọng: Bạn đã bỏ sót Chrome Built-in Translation API
Đây là điểm **lớn nhất** mà phân tích của bạn chưa đề cập, và nó thay đổi toàn bộ bài toán API từ phía tôi.
**[CONFIRMED via Tavily Search: Chrome Built-in Translator API]**
Chrome (từ phiên bản 138+) đã tích hợp **Translator API và LanguageDetector API** chạy hoàn toàn trên thiết bị (on-device), sử dụng mô hình Gemini Nano:
// Phát hiện ngôn ngữ - không cần API key, không gửi data ra ngoài
const detector = await window.LanguageDetector.create();
const results = await detector.detect(selectedText);
// results.detectedLanguage → "en", "vi", "ja", ...
// Dịch - hoàn toàn offline sau khi tải model lần đầu
const translator = await window.Translator.create({
  sourceLanguage: results.detectedLanguage,
  targetLanguage: 'vi',
});
const translated = await translator.translate(selectedText);
**Tác động kiến trúc của Built-in API:**
> **Hệ quả:** Với Built-in API, toàn bộ bài toán "chọn API nào để cân bằng chi phí" trở nên **vô nghĩa** – vì không có chi phí. Đây là lựa chọn lý tưởng cho extension miễn phí.
**Nhưng có một "catch" lớn:** Built-in API chỉ hoạt động khi code chạy trong **page context** (có `window.Translator`). Background script/service worker **không có** quyền truy cập API này. Điều này tạo ra một nghịch lý với kiến trúc `contextMenus` của bạn:
- `contextMenus` → background script nhận text → **nhưng background script không dùng được Built-in API**
- Muốn dùng Built-in API → phải inject script vào page context → **đúng thứ bạn muốn tránh cho native PDF viewer**
Đây là điểm mâu thuẫn kiến trúc then chốt mà tôi muốn bạn cân nhắc.

**Điểm mấu chốt:** Nếu bạn chọn `contextMenus` làm primary path (như đề xuất), bạn **buộc phải dùng Cloud API**. Ngược lại, nếu muốn dùng Built-in API (miễn phí, bảo mật tuyệt đối), bạn **buộc phải inject script** vào page context – ít nhất là cho các trang web thông thường và PDF.js.

### 🚩 Các điểm còn thiếu trong phân tích của bạn:
1. **Bạn chưa chọn translation API cụ thể** — "kết nối với Google Translate hoặc các trình dịch khác" là quá mơ hồ. Cần chốt:

   - **LibreTranslate self-hosted**: Hoàn toàn miễn phí, tự chủ.
2. **Bạn chưa đề cập đến cơ chế hiển thị kết quả**: Context menu click → xong? Dịch xong hiển thị ở đâu?
   - `chrome.notifications` (dễ nhưng intrusive)
   - Popup badge (cần click icon để xem)
   - Side panel API (`chrome.sidePanel`) — đây là cách tốt nhất cho Manifest V3
3. **Bạn chưa xử lý vòng đời Service Worker**: SW tắt sau ~30s. Nếu gọi API dịch mất >30s (text dài), SW sẽ chết giữa chừng. Cần dùng `chrome.storage.local` làm communication bridge.
4. **Thiếu cơ chế cache**: Dịch cùng một text nhiều lần = waste. Cần LRU cache với `chrome.storage.local`.

### ⚠️ Bạn thiếu một điểm quan trọng: Content script ISOLATED world vs `window.Translator`
**[CONFIRMED via Chrome Docs + StackOverflow]**
Content scripts trong Chrome chạy trong **isolated world** – nghĩa là code content script **không thể** truy cập biến/API của page context (MAIN world). `window.Translator` là API của MAIN world.
┌─────────────────────────────────────────────┐
│  PAGE CONTEXT (MAIN World)                   │
│  window.Translator.create()  ← API ở đây     │
│  window.LanguageDetector.create()            │
├─────────────────────────────────────────────┤
│  CONTENT SCRIPT (ISOLATED World)             │
│  window.Translator → ❌ undefined            │
│  Có thể: DOM access, chrome.runtime.*        │
└─────────────────────────────────────────────┘
Điều này ảnh hưởng trực tiếp đến Path A của bạn:
**Giải pháp cho Path A (PDF.js / web pages):** Cần thêm một **MAIN world injection bridge**:
Content script phát hiện selection
        │
        ▼ gửi text qua chrome.runtime.sendMessage()
Background service worker
        │
        ▼ inject MAIN world script via chrome.scripting.executeScript({world: "MAIN"})
MAIN world script
        │
        ▼ gọi window.Translator.create() + translate()
        ▼ post kết quả về content script qua window.postMessage()
Content script
        │
        ▼ nhận translation → hiển thị tooltip
Điều này **khả thi** nhưng phức tạp hơn đáng kể so với gọi REST API từ background.

#### 1️⃣ Vấn đề chất lượng text từ PDF.js (Path A')
Khi content script gọi `window.getSelection().toString()` trên một PDF.js viewer, text thường bị nhiễm artifacts:
**Nguyên nhân:** PDF.js text layer render từng ký tự trong `<span>` riêng:
